# Freedom Hackathon 2026 - Churn Intelligence

## 1) Цель и бизнес-постановка

Задача: не просто предсказать, кто уйдет, а сделать удержание управляемым и экономически эффективным.

Мы реализовали `Churn Intelligence`:
- оцениваем вероятность оттока пользователя;
- учитываем ценность пользователя (`pLTV/Freedom Score proxy`);
- переводим прогноз в 4 операционных сегмента с конкретными действиями.

Итоговая цель для бизнеса: максимизировать retention uplift при ограниченном маркетинговом бюджете.

---

## 2) Какие данные использовали и почему

Использованные источники из PostgreSQL:
- `feature_store` (базовые агрегированные признаки по активности/транзакциям);
- `events` (поведение в приложении, типы и статусы событий);
- `transactions` (финансовая активность, объемы, частота, отказные операции);
- `acquisition` (канал привлечения);
- `pltv_predictions` (оценка ценности пользователя);
- `user_segments` (ценностные сегменты);
- `users` (демография, для EDA и части признаков).

Почему не подавали "все подряд":
- сырые логи (`events`, `transactions`) очень шумные и тяжелые;
- для модели churn использовали нормализованный слой `feature_store` + целевые допризнаки;
- высоко-кардинальные и нестабильные поля (например, часть сырого статуса) не включались напрямую как ключевые признаки;
- `acquisition`, `pltv_predictions`, `user_segments` были дедуплицированы по `user_id` перед merge.

## 3) Подготовка данных и контроль качества

### 3.1 Data quality checks

Проверили:
- диапазоны дат по таблицам;
- распределения категорий/статусов/каналов;
- долю пустых/аномальных значений;
- наличие дубликатов по `user_id` в таблицах-справочниках.

### 3.2 Критичные сложности и как исправили

1. **Row explosion при merge**  
`pltv_predictions` и `user_segments` имели несколько строк на одного пользователя.  
Решение: SQL-дедупликация:
- для `pltv_predictions` брали `MAX(pltv_predicted)` по `user_id`;
- для `user_segments` брали приоритетный сегмент (VIP > Medium > Loss-Making);
- для `acquisition` брали один канал на пользователя.

2. **Некорректный test split в ранней версии**  
Первый временной split дал перекос класса в тесте.  
Решение: стратифицированный split для стабильной оценки качества на текущем snapshot-датасете.

3. **Интерпретация сегментов в интерфейсе**  
Streamlit `delta` визуально выглядел как "рост", хотя это была доля сегмента.  
Решение: заменили на явный текст "Доля сегмента: X%".

## 4) Таргет (label) и логика постановки

Таргет churn:
- `churn_label = 1`, если последняя активность пользователя (последнее событие или транзакция) старше `30` дней относительно reference date;
- иначе `0`.

Reference date использовали как конец доступного горизонта данных.

Почему так:
- соответствует брифу ("inactive for 30+ days");
- бинарная постановка с понятной интерпретацией для бизнеса и метрик (AUC/Precision/Recall/F1).

---

## 5) Feature engineering (без "воды", только то, что реально работает)

Базовые признаки из `feature_store`:
- rolling windows: `1d/3d/7d/14d` по транзакциям и событиям;
- объемы, частоты, активные дни, ошибки/неуспехи;
- продуктовые бинарные флаги (flights, tickets, groceries, media);
- `proxy_ltv_90d`, возраст, пол.

Добавленные производные признаки:
- **поведенческие тренды**:
  - `tx_trend = tx_cnt_7d / (tx_cnt_14d - tx_cnt_7d + eps)`
  - `evt_trend`, `vol_trend`
- **friction/frustration**:
  - `error_evt_rate`
  - `failed_tx_rate`
  - `completed_evt_rate`
- **цифровой профиль**:
  - `digital_ratio` (purchase vs p2p/atm mix)
- **cross-sell depth**:
  - `product_depth` как сумма продуктовых флагов (вместо нестабильных ratio).
- **acquisition proxy**:
  - `is_paid_channel` (paid vs non-paid).

Почему так:
- churn сильнее всего объясняется динамикой поведения и friction-сигналами;
- тренды и rates устойчивее сырых абсолютов;
- бинарные продуктовые флаги интерпретируемы и менее шумные, чем сложные отношения.

## 6) Модель и обучение

Выбранная модель: `CatBoostClassifier`.

Почему CatBoost:
- хорошо работает на табличных данных;
- устойчив к нелинейностям и взаимодействиям признаков;
- поддерживает категориальные признаки;
- дает качественную базу для SHAP-интерпретации.

Параметры обучения (ключевые):
- `iterations=1000`
- `learning_rate=0.05`
- `depth=6`
- `eval_metric=AUC`
- `early_stopping_rounds=50`
- `class_weights={0:1, 1:2}` для усиления churn-класса.

Split:
- `train_test_split(..., stratify=y, random_state=42)`.

## 7) Что происходило во время обучения (фактически)

Финальный запуск:
- датасет после clean merge: `373,304` пользователей;
- churn rate: `34.30%`;
- train: `298,643`, test: `74,661`.

Train log:
- итерации прошли полностью до `1000`;
- `bestTest (AUC)` достигнут на финальных итерациях;
- переобучение контролировалось early stopping и регуляризацией.

Итоговые метрики:
- `AUC-ROC = 0.8086`
- `F1 = 0.6538`
- `Precision = 0.5555`
- `Recall = 0.7942`

Интерпретация:
- модель хорошо ранжирует риск (`AUC > 0.8`);
- высокий `Recall` полезен для retention-задачи (лучше найти больше риска и потом приоритизировать бюджетом).

## 9) Сегментация для действий (operational layer)

Мы не остановились на вероятности оттока, а добавили операционный слой:

Сегменты:
- `Persuadables`: высокий churn риск + высокая ценность;
- `Sure Things`: низкий churn риск + высокая ценность;
- `Sleeping Dogs`: высокий churn + высокий friction (лучше не трогать);
- `Lost Causes`: высокий churn + низкая ценность;
- `Low Value Stable`: низкий churn + низкая ценность.

Это дает конкретный action policy:
- бюджет удержания фокусируем на `Persuadables`;
- `Sure Things` -> upsell;
- `Sleeping Dogs` -> no-contact;
- `Lost Causes` -> минимальные затраты, feedback loop.

## 10) Почему решение сильное и в чем уникальность

1. **Actionable, а не только predictive**  
Не просто "кто уйдет", а "на кого тратить retention-бюджет и как".

2. **Связка риска с ценностью пользователя**  
Решение оптимизирует не количество контактов, а бизнес-эффект.

3. **Explainable by design**  
SHAP-пояснение факторов для product/marketing команд.

4. **Готово к демонстрации и внедрению**  
Есть рабочий Streamlit dashboard + выгрузка `users_scored.csv` для CRM-кампаний.

## 11) Практические рекомендации бизнесу

1. Запускать retention-кампании по `Persuadables` с приоритетом по `pLTV`.
2. Для `Sleeping Dogs` снизить коммуникационное давление, сначала исправлять UX/error hotspots.
3. Для `Sure Things` переключать фокус с удержания на cross-sell/upsell.
4. Пересчитывать scoring регулярно (например, ежедневно/еженедельно) и отправлять в CRM.
5. В следующей итерации добавить:
   - temporal validation на нескольких срезах времени;
   - uplift-эксперименты (A/B) для валидации эффективности офферов;
   - event sequence embeddings для усиления качества модели.
